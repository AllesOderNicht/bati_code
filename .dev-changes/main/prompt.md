# Prompt

- Branch: `main`

## Context
- `bati` 是一个类似 `MBTI` 的趣味测试网站，但结果映射为“你更像哪家大厂的人设 / 气质”。
- 首版产品目标不是做严肃职业测评，而是做可传播、可分享、带品牌联想的轻量趣味测试。
- 已确认技术栈为 `Vite + React + TypeScript`，实现形态为纯前端可交互网站。
- 已确认算法方向为规则加权型，即每道题为一个或多个公司画像贡献权重，再汇总得到结果。
- 已确认首版公司池不是小样本验证，而是直接覆盖“国内前三十 + 国外前十”的全量目标公司池。
- 已确认结果页只展示 `Top1` 主结果，不做完整排行榜式展示。
- 已确认结果内容深度为轻量版，仅包含公司标签、1 段人格描述、3-5 个关键词。
- 当前原始需求中“展示候选公司概率分布”和“首版只展示 Top1”之间存在产品表达冲突，需要在解释性和简洁性之间做第二轮澄清。
- 当前全量公司池、正向玩梗约束、轻量内容深度三者叠加后，内容生产方式与公司准入治理方式会直接决定首版是否可控。

## Requirements

### Functional Requirements
- 提供一个可直接交互的前端测试网站，完成从答题到结果展示的完整闭环。
- 题目围绕性格、生活方式、价值观、工作方式等轻量维度设计，不做严肃心理测评。
- 题目需要能够区分不同公司的“梗人格”与气质归属，并为规则加权算法提供输入。
- 首版目标题量约为 30 题，但最终题型和交互流程仍受第二轮产品范围决策影响。
- 结果页默认只输出 1 个主结果公司，不做完整榜单型结果页。
- 结果内容采用轻量输出：公司标签、1 段人格描述、3-5 个关键词。
- 文案需要围绕公司公开形象、互联网正向梗和人格联想生成，强调趣味性、传播性和可分享性。
- 公司池目标覆盖中国主流大厂与代表性海外科技公司，首版以“国内前三十 + 国外前十”为覆盖边界。

### Non-Functional Requirements
- 技术实现必须基于 `Vite + React + TypeScript`。
- 算法必须保持可解释性，至少能说明为什么用户会命中某一类公司画像。
- 内容表达必须积极、轻松、幽默，不允许攻击、侮辱或恶意刻板印象。
- 结果表达应强调“你像这家公司的人设气质”，而不是“你应该去这家公司工作”。
- 首版范围需要优先保证可落地，而不是追求复杂动画、重内容深度或高维护成本生产模式。

## Scope

### In Scope
- 一个前端答题站点及其基础交互流程。
- 一套规则加权型结果判定机制。
- 一套面向大厂画像的轻量结果文案体系。
- 首版全量公司池的结果映射能力。
- 正向玩梗语气与基础可分享结果展示。

### Out of Scope
- 严肃人格测评、职业能力评估、招聘导向结论。
- 对公司或群体的负面标签化表达。
- 首版的深度内容卡片、长篇公司画像解析、复杂榜单页。
- 未经治理直接覆盖所有可能引发争议的公司梗内容。

## Key Decisions

### Checkpoint 1 Decisions
- Q: 首版技术栈使用什么？ → A: `Vite + React + TypeScript` → Decision: 首版按现代前端单页应用思路实现，避免引入服务端复杂度。
- Q: 首版公司池范围如何定？ → A: 直接覆盖“国内前三十 + 国外前十”的全量公司池目标 → Decision: 不采用先少量样本公司试水的策略，首版即面向大规模公司画像配置。
- Q: 结果页展示形式如何定？ → A: 只展示 `Top1` 主结果 → Decision: 结果页不走排行榜视觉，需要用更轻量的方式完成解释性表达。
- Q: 结果内容深度如何定？ → A: 轻量版（公司标签、1 段人格描述、3-5 个关键词） → Decision: 首版优先保证命中感和传播性，不做长文案与多模块深度解析。
- Q: 算法方式如何定？ → A: 规则加权型 → Decision: 不走模型推断或复杂推荐系统，优先选择可控、可调、可解释的分值映射。

### Checkpoint 2 Divergences And Pending Clarifications
- Divergence: 原始需求曾要求展示“各公司或候选公司的概率分布”，但 Checkpoint 1 已明确结果页只展示 `Top1` 主结果。 → Pending: 是否仍需保留少量相近候选或置信提示来补足解释性。
- Code-Intent Conflict: 首版要直接覆盖“国内前三十 + 国外前十”，但结果内容又被限制为轻量、正向、低风险表达。 → Pending: 是否接受“统一算法模板 + 公司画像配置”的规模化内容生产方式，以及是否对公司池启用人工白名单。
- Missing Decision: 题目形式、答题流程、移动端优先级、视觉风格会直接影响题量、页面数量、动效成本和首版工期。 → Pending: 需要明确首版交互复杂度与视觉野心。

### Checkpoint 2 Follow-up Questions
**1. 关于“首版全量公司池”与“轻量内容深度”的实现冲突：如果首版要一次性覆盖国内前三十 + 国外前十，而结果内容又只做轻量版，你是否接受采用“同一算法模板 + 不同公司画像配置”的规模化生产方式，并配合公司白名单治理？**
   A) 接受。统一题库维度和打分模板，每家公司主要靠画像配置、标签和文案参数区分；同时用人工白名单排除不适合玩梗或素材不足的公司。
   B) 半接受。算法模板可以统一，但公司要先归成少数几类画像原型，再在原型上做公司级微调；白名单只用于排除高风险公司。
   C) 不接受。每家公司都应有明显差异化题目映射和结果文案，即使这会显著增加首版内容生产成本；白名单只做极少量兜底。
   D) 其他（请说明）

**2. 关于“只展示 Top1”与“仍需可解释性”的冲突：当前方向是结果页只放 1 个主结果，你更希望首版怎样补足解释感？**
   A) `Top1` 主结果 + 2 个相近公司提示（仅显示名称/相似度短条），不展开完整榜单。
   B) 只显示 `Top1`，但补充“你为什么像它”的关键词和命中维度解释，不展示任何其他公司。
   C) `Top1` 主结果 + 一个模糊置信区间提示，例如“你的结果很稳定 / 边界感较强”，不直接点名第二名第三名。
   D) 其他（请说明）

**3. 关于首版范围控制：题目形式、用户流程、移动端优先程度和视觉风格会直接决定工作量，你更倾向哪种首版方案？**
   A) 移动端优先。单题一步式流程或滑卡式答题，视觉轻快有梗，但动效控制在基础范围内。
   B) 通用响应式优先。标准分页答题流程，视觉简洁清爽，优先快速上线和稳定实现。
   C) 品牌感优先。允许更强视觉风格和更明显的结果页包装，即使题目流程和页面动效成本更高。
   D) 其他（请说明）

## Notes
- 该产品的核心价值不是“真实职业测评”，而是“有梗、有分享欲、有品牌联想的趣味测试”。
- 当前 `prompt.md` 已整合原始需求、Checkpoint 1 已确认决策，以及 Checkpoint 2 待确认问题；后续应以此文件作为单一事实来源继续推进。
