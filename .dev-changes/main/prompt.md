# Prompt

- Branch: `main`

## Context
- `bati` 是一个类似 `MBTI` 的趣味测试网站，但结果映射为“你更像哪家大厂的人设 / 气质”，定位是轻量、可传播、带品牌联想的娱乐化测试，而不是严肃职业测评。
- 技术栈已锁定为 `Vite + React + TypeScript`，首版实现形态为纯前端单页应用，不引入服务端依赖作为首发前提。
- 当前仓库仍处于规划阶段，实际仅存在 `.dev-changes/main` 下的变更文档，尚未初始化 `Vite` 工程、业务源码、测试基础设施或构建配置。
- 首版公司池目标为“国内前三十 + 国外前十”，不是小样本试水，因此必须采用可规模化维护的数据结构和内容生产方式。
- 结果页只展示 `Top1` 主结果，解释性通过“你为什么像它”的命中关键词与维度说明补足，不展示候选公司列表或概率分布。
- 结果内容深度保持轻量，只输出公司标签、1 段人格描述和 3-5 个关键词，强调命中感、分享感和轻松梗感。
- 题目交互已锁定为统一四选一；整体体验采用移动端优先、单题一步式或滑卡式答题，视觉轻快有梗，但动效只做基础层。
- 内容生产接受“统一算法模板 + 不同公司画像配置”的方式，并配合人工白名单治理，确保全量公司池在首版内仍可控、可审查、可迭代。
- 公司命名策略采用中文优先，结果页默认展示中文公司名，不在首版实现复杂别名字典或多语言命名策略。

## Requirements

### Functional Requirements
- 提供一个完整的前端测试网站闭环，覆盖首页引导、答题流程、结果展示和基础回流能力。
- 题目围绕性格、生活方式、价值观、工作方式等轻量维度设计，统一使用四选一输入形式，保证节奏快、理解成本低、适合移动端连续作答。
- 答题交互采用单题一步式或滑卡式流程，用户每一步只处理一题，需有清晰的进度反馈、已选状态和下一步动作。
- 规则加权算法需要根据用户答案计算各公司画像分值，输出唯一 `Top1` 结果，并生成对应的命中解释。
- 结果页必须包含中文公司名、公司标签、1 段人格描述、3-5 个关键词，以及“你为什么像它”的关键词和命中维度说明。
- 公司配置体系必须拆分为分层 schema，至少覆盖公司基础资料、算法权重配置、结果文案配置、品牌别名 / 治理信息四类结构。
- 首版需支持覆盖“国内前三十 + 国外前十”的目标公司池，并通过白名单机制控制可上线公司范围。
- 结果页需要具备“可分享感”，但首版只做到视觉和内容表达层面的分享氛围，不做海报生成，也不做复制文案按钮。

### Non-Functional Requirements
- 技术实现必须基于 `Vite + React + TypeScript`，并保持前端结构清晰、配置可维护、适合后续继续扩题和扩公司。
- 算法必须可解释、可调试、可审阅，不能是黑盒结果；结果解释应能追溯到题目命中维度和加权逻辑。
- 内容表达必须积极、轻松、正向、有梗，但不得攻击公司、嘲讽从业者、暗示优劣排名或制造负面刻板印象。
- 结果表达必须强调“你像这家公司的人设气质”，不能被误读为“你适合去这家公司工作”或“你应该加入这家公司”。
- 移动端体验优先，视觉可以采用圆润、轻盈、偏玩具感的柔和风格，但应控制动效复杂度，保障首版实现成本和性能。
- 首版必须优先保证可落地和内容治理可控，不引入复杂分享系统、重型动画、服务端账号体系或高维护成本后台。

## Scope

### In Scope
- 从零初始化 `Vite + React + TypeScript` 前端工程并建立可运行的答题站点骨架。
- 统一四选一题库模型、答题状态管理、基础移动端交互和进度反馈。
- 规则加权型评分引擎、命中解释逻辑和 `Top1` 结果输出。
- 面向全量公司池的分层配置 schema、内容装配方式和白名单治理机制。
- 轻量结果页、分享感表达、正向玩梗文案边界和基础验证能力。

### Out of Scope
- 严肃人格测评、职业能力评估、招聘导向推荐或岗位匹配结论。
- 展示完整候选公司榜单、第二名第三名、概率分布图或复杂相似度排行页。
- 海报生成、复制分享文案按钮、登录系统、服务端存储、用户历史记录。
- 复杂别名字典、多语言命名系统、深度公司百科卡片、长篇解析内容。
- 未经白名单治理直接上线所有公司画像，或使用可能引发负面争议的尖锐玩梗表达。

## Key Decisions

### Checkpoint 1 Decisions
- Q: 首版技术栈使用什么？ → A: `Vite + React + TypeScript` → Decision: 首版采用现代前端单页应用方案，避免引入服务端复杂度。
- Q: 首版公司池范围如何定？ → A: 直接覆盖“国内前三十 + 国外前十”的全量公司池目标 → Decision: 首版即面向大规模公司画像配置，不走少量样本试水。
- Q: 结果页展示形式如何定？ → A: 只展示 `Top1` 主结果 → Decision: 结果页不做排行榜视觉，解释性通过补充模块实现。
- Q: 结果内容深度如何定？ → A: 轻量版（公司标签、1 段人格描述、3-5 个关键词） → Decision: 首版优先保证命中感和传播性，不做长文案与多模块深度解析。
- Q: 算法方式如何定？ → A: 规则加权型 → Decision: 不走模型推断或复杂推荐系统，优先选择可控、可调、可解释的分值映射。

### Checkpoint 2 Decisions
- Q: 面对“全量公司池 + 轻量内容深度”的实现冲突，内容生产策略如何定？ → A: 接受“统一算法模板 + 不同公司画像配置”的规模化生产方式，并配合人工白名单治理 → Decision: 首版采用统一题库维度和评分模板，以公司级配置驱动差异化输出，并通过白名单控制上线范围。
- Q: 面对“只展示 Top1”与“仍需可解释性”的冲突，结果解释模式如何定？ → A: 只显示 `Top1`，但补充“你为什么像它”的关键词和命中维度解释，不展示其他公司 → Decision: 结果页专注单一主结果，同时增强解释区而非扩展候选区。
- Q: 首版交互和视觉范围如何定？ → A: 移动端优先，单题一步式流程或滑卡式答题，视觉轻快有梗，但动效保持基础 → Decision: 首版优先优化移动端节奏感和轻巧视觉，不投入高成本复杂动画。

### Checkpoint 3 Decisions
- Q: 公司配置体系如何拆分？ → A: 分层 schema，将“公司基础资料”“算法权重配置”“结果文案配置”“品牌别名 / 治理信息”拆成独立结构 → Decision: 业务数据模型按职责分层，便于批量维护、审核和算法调优。
- Q: 题目输入形式如何统一？ → A: 统一四选一 → Decision: 交互层和题库层都按单选四选项模型设计，减少组件分叉和内容维护成本。
- Q: 首版分享能力边界如何定？ → A: 只做结果页分享感，不做海报生成，也不做复制文案按钮 → Decision: 传播性通过结果页视觉、标题和轻量文案实现，不扩张到复杂分享工具链。
- Q: 公司命名策略如何定？ → A: 中文优先，结果页默认使用中文公司名展示，不做复杂别名字典策略 → Decision: 首版展示口径统一，用最小命名治理实现足够一致的用户认知。
