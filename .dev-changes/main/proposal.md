# Proposal: BATI 大厂人格测试首版

## Goal
打造一个移动端优先的趣味测试网站，让用户通过一组轻量四选一问题，得到“你更像哪家大厂的人设 / 气质”的唯一主结果。首版重点不是测得多准，而是要在可解释、可传播、可治理的前提下，提供足够强的命中感和分享欲。

## Background
当前项目仍处于规划阶段，尚未落地任何前端工程或内容配置系统，因此首版需要同时解决三件事：从零搭建可运行的交互站点、建立可规模化维护的题库与公司配置模型、把结果页做成既轻量又有“被说中”感觉的内容体验。由于公司池一开始就要覆盖“国内前三十 + 国外前十”，系统设计必须优先考虑配置化、审核成本和文案边界，而不是只为少量样本临时拼装。

## User Stories
- 作为一个在手机上刷到测试链接的用户，我希望几分钟内完成答题，并得到一个明确、有趣、容易理解的结果。
- 作为一个对互联网品牌梗熟悉的用户，我希望结果足够像某家公司的气质，但不要带有恶意和冒犯。
- 作为内容维护者，我希望题目、算法权重、结果文案和治理信息能分层维护，不需要每新增一家公司都重做整套逻辑。
- 作为产品 owner，我希望首版在不做服务端和复杂分享工具的情况下，仍具备基础传播性和后续扩展空间。

## First Release User Flow
1. 用户进入首页，看到一句清晰的产品定位和开始答题入口。
2. 用户按单题一步式流程完成四选一答题，页面持续反馈当前进度。
3. 用户提交最后一题后，系统运行规则加权算法，结合白名单与命名治理规则产出唯一 `Top1` 结果。
4. 结果页展示中文公司名、公司标签、1 段人格描述、3-5 个关键词，以及“你为什么像它”的命中解释。
5. 用户在结果页获得足够的截图分享感，但首版不提供海报生成和复制文案能力。

## Question Model And Interaction Principles
- 所有题目统一为四选一，避免混用多种输入形式导致的认知跳变和组件复杂度膨胀。
- 每题只表达一个轻量判断场景，例如决策偏好、协作风格、信息处理方式、生活方式倾向或价值感来源。
- 每个选项可以同时映射到若干“命中维度”和若干公司权重，保证算法既可解释又能保留公司间差异。
- 首版交互坚持“一屏一题、一题一决策”，优先保证移动端手指操作和节奏感，不追求复杂动效编排。
- 题目文案要先满足“看得懂、选得快”，再追求梗感；避免过度黑话、过度圈层梗和需要行业前置知识的表达。

## Technical Approach
首版采用配置驱动的纯前端架构，核心模块建议如下：

- `QuestionBank`：管理四选一题目、选项文案、维度映射和展示顺序。
- `CompanyProfileRegistry`：按分层 schema 存放公司基础信息、算法权重配置、结果文案配置和治理元数据。
- `ScoringEngine`：根据用户答案聚合权重，输出排序后的内部候选集，再经过治理规则得出最终 `Top1`。
- `ExplanationExtractor`：从用户命中最高的维度、关键词和选项轨迹中提取“为什么像它”的解释素材。
- `GovernanceRegistry`：处理白名单、禁用状态、风险备注、中文优先展示名和最小别名兼容。
- `ResultComposer`：把算法结果转换成适合结果页展示的轻量 view model。

推荐的接口边界如下：

- `scoreQuiz(answers, questions, companyProfiles) => RankedCompanyResult[]`
- `resolvePrimaryResult(rankedResults, governanceRules) => PrimaryCompanyResult`
- `buildExplanation(primaryResult, answers, questions) => ExplanationBlock`
- `composeResultView(primaryResult, explanation, copyConfig) => ResultViewModel`

### Rule-Based Weighting Strategy
- 每道题的每个选项同时携带维度权重和公司权重，两者共同参与累计。
- 维度权重用于解释层，回答“你像它的原因是什么”；公司权重用于结果层，回答“最终更像哪一家”。
- 评分流程应先完成答案聚合，再执行白名单过滤、禁用公司剔除和稳定性判断，最后生成唯一主结果。
- 当多个公司分数接近时，首版不显示第二名第三名，而是在内部使用稳定 tie-break 规则，例如优先比较核心维度命中数、再比较关键题命中权重、最后按显式治理优先级兜底。
- 结果解释不展示完整候选排序，而是输出若干命中维度和关键词，让用户感觉“被看见”而不是“被排名”。

## Layered Company Schema Design
建议将公司配置拆为四层，并保持相互解耦：

- `CompanyBaseProfile`
  - 唯一标识
  - 中文展示名
  - 品牌标签
  - 地域和类别等轻量元信息
- `CompanyScoreProfile`
  - 维度权重偏好
  - 关键题加权规则
  - tie-break 辅助权重
- `CompanyCopyProfile`
  - 结果标题
  - 1 段人格描述
  - 3-5 个关键词
  - 可复用的解释模板片段
- `CompanyGovernanceProfile`
  - 是否在白名单内
  - 风险备注
  - 中文优先显示策略
  - 最小别名支持

这种拆法的价值在于：算法调优、文案改写、品牌治理和基础资料维护可以分别进行，不会互相污染。

## Whitelist And Governance Strategy
- 首版必须采用人工白名单作为上线门槛，未进入白名单的公司即使配置存在，也不能参与正式结果输出。
- 治理重点不是审查“是否好笑”，而是审查“是否正向、是否容易误伤、是否有足够公开形象支撑”。
- 公司别名策略保持最小化，只处理必要的内部兼容和数据映射，前台一律以中文优先名展示。
- 文案中禁止使用负向职场暗示、薪资讽刺、内卷羞辱、鄙视链表达和疑似招聘导向语句。
- 对高争议公司或素材不足公司，应允许先存在配置草稿，但默认不进入白名单。

## Result Copy Structure And Tone Boundary
结果页建议固定为以下结构：

1. 主标题：`你像 <中文公司名> 派`
2. 标签区：2-3 个品牌气质标签
3. 人格描述：1 段轻量、有画面感、正向玩梗的描述
4. 关键词区：3-5 个短词，强调节奏、风格、处事方式
5. 解释区：若干命中维度和对应短句，回答“你为什么像它”

正向玩梗边界：

- 可以调侃工作方式和气质风格，但不能攻击公司道德、讽刺员工价值或制造优劣鄙视链。
- 可以写“节奏快、产品感强、执行力拉满”这类中性偏正向表达，不能写“压榨、卷王、福报”这类高风险词。
- 可以制造分享感和识别度，但不能让结果页变成品牌点评榜或就业建议页。

## Visual And Mobile Strategy
- 体验优先级以手机竖屏为主，桌面端只做自然兼容，不反向主导信息布局。
- 视觉气质建议走圆润、轻盈、略带玩具感的风格：大圆角、柔和表面、轻量层次阴影、暖色或轻饱和配色，而不是锐利或过于“企业官网化”的样式。
- 按钮、卡片、选项态和结果标签都应共享一套统一设计 token，保证从首页到结果页的调性一致。
- 动效保持基础，只做必要的切题反馈、选中反馈和结果进入过渡，同时尊重低动效偏好。
- 分享感主要来自结果卡片的排版、标题语气和关键词视觉，而不是额外的海报或下载能力。

## Product Goals And Non-Goals
### In Scope (MUST)
- MUST: 提供从首页、答题到结果的完整首版流程。
- MUST: 所有题目使用四选一交互，并适配移动端单题一步式体验。
- MUST: 用规则加权算法输出唯一 `Top1` 结果，并附带命中解释。
- MUST: 使用分层 schema 承载全量公司池配置，并配合人工白名单治理。
- MUST: 结果页输出轻量文案、标签、关键词和正向玩梗表达。

### Out of Scope (MUST NOT)
- MUST NOT: 展示完整候选榜单、概率分布或多结果对比。
- MUST NOT: 上线海报生成、复制文案按钮、登录系统和历史记录。
- MUST NOT: 输出带攻击性、争议性或招聘导向的结论。
- MUST NOT: 在首版引入复杂别名字典和多语言命名系统。
- MUST NOT: 为追求视觉包装而堆叠高成本动画和重运营组件。

### Optional (MAY)
- MAY: 在后续版本补充更细的维度体系和更丰富的解释模板。
- MAY: 为桌面端加入更完整的响应式细节，但不改变移动端优先策略。
- MAY: 在白名单机制稳定后扩展更多公司或更精细的品牌原型分组。

## Validation Strategy
- 使用配置校验确保题目、选项、公司基础资料、文案配置和治理字段完整且结构正确。
- 使用算法单元测试和快照测试验证打分稳定性、tie-break 顺序和解释提取逻辑。
- 使用人工内容评审清单检查结果文案是否越过正向玩梗边界。
- 使用移动端优先的交互冒烟验证，确认单题流程、进度反馈、结果展示和回流行为无阻塞。
- 使用少量代表性答案样本做结果 sanity check，避免出现所有人都高频命中同一家公司的偏置现象。

## Risks
- 全量公司池首版即上线，内容和配置工作量大。缓解方式是坚持分层 schema 和统一模板，并以白名单控制上线批次。
- 统一算法模板可能让结果趋同。缓解方式是保留公司级权重差异、关键题加权和解释模板差异。
- 玩梗文案容易越界。缓解方式是建立明确禁用词和人工审核门槛，先保守再放开。
- 当前仓库没有任何工程基础。缓解方式是把工程搭建列为独立优先阶段，先建立可运行骨架和测试基线。
